# Архитектура

Документ описывает внутреннюю модель проекта. README остается пользовательским и продуктовым, а технические детали живут здесь.

## Основная идея

Проект разделяет два сценария:

- разовые занятия;
- постоянное расписание.

Разовые занятия хранятся как конкретные записи. В сценарии разовых занятий пользователь может выбрать одну дату или несколько дат одним календарем; для каждой выбранной даты создается отдельная запись в `lessons`.

Постоянное расписание поддерживает два варианта:

- занятия по дням недели;
- выбранные конкретные даты.

Занятия по дням недели хранятся как правило, из которого бот генерирует ближайшие занятия и напоминания.

Конкретные выбранные даты не являются правилом повторения, поэтому сохраняются как обычные записи в `lessons`.

Такой подход не копирует десятки одинаковых уроков в базу и оставляет место для будущих исключений: отмены или переноса одного конкретного занятия.

## Таблицы

### `teachers`

Репетиторы бота. Каждый пользователь, который запускает обычный `/start`, получает свою запись в этой таблице.

Поле `setup_completed_at` отмечает, что репетитор уже прошел короткий первый путь: добавил ученика и создал первое занятие или постоянное расписание. Благодаря этому завершение онбординга показывается один раз.

### `students`

Ученики репетитора. Ученик попадает в систему только через invite-ссылку.

### `invites`

Invite-ссылки для подключения учеников. Владелец не может использовать собственную ссылку.

### `lessons`

Разовые занятия и будущие исключения из постоянного расписания.

Ключевые поля:

- `schedule_rule_id` — ссылка на правило, если это исключение из регулярного расписания;
- `starts_at` — фактическое время занятия;
- `original_starts_at` — исходное время, если занятие перенесено;
- `status` — состояние занятия.

Статусы:

- `active` — занятие активно;
- `cancelled` — занятие отменено.

Перенос определяется не отдельным статусом, а наличием `original_starts_at`.

### `schedule_rules`

Постоянное расписание.

Ключевые поля:

- `student_id`;
- `weekdays`;
- `lesson_time`;
- `start_date`;
- `end_date`;
- `reminders`;
- `is_active`.

Формат `weekdays`: числа `0-6`, где `0` — понедельник, `6` — воскресенье. Например, `1,3` означает вторник и четверг.

Формат `reminders`: строка через запятую, например `day,hour,10m`.

### `reminders`

Напоминания для разовых занятий. Для них заранее создаются записи с `remind_at`.

### `reminder_logs`

Журнал отправки напоминаний для регулярных занятий.

Первичный ключ:

```text
(occurrence_key, kind)
```

Это защищает от повторной отправки после перезапуска worker'а.

## Напоминания

Worker раз в минуту проверяет два источника:

- разовые занятия через `reminders`;
- постоянное расписание через `schedule_rules`.

Для регулярных занятий worker генерирует ближайшие экземпляры в окне проверки и смотрит, отправлялось ли уже уведомление для пары `occurrence_key + kind`.

## Ограничения MVP

Сейчас проект сознательно не реализует всю сложность Google Calendar:

- нет RRULE;
- нет изменения "этого и всех следующих";
- нет сложных исключений;
- нет синхронизации календарей.

Ближайшие логичные шаги:

- отмена одного регулярного занятия;
- перенос одного регулярного занятия;
- редактирование правила расписания.
